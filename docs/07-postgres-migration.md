# 07 — PostgreSQL Migration Runbook

Prepared during Phase 0. Execute at the start of the next work session (it is the first
step of Phase 1, since the catalog tables require `Json` columns that Prisma does not
support on SQLite).

**Target schema is ready:** [`prisma/postgres/schema.prisma`](../prisma/postgres/schema.prisma)
— all current models (with the three Phase 0 migrations folded in), string-JSON fields
upgraded to real `Json`, booking extensions (`details`, `BookingItem`), and the full
catalog table set (`ServiceCategory`, `Subcategory`, `Service`, `ServicePackage`,
`ServiceAddon`, `PriceRule`) for the service abstraction in doc 04.

## Why reset-and-reseed (not data migration)

- Production (Netlify) currently runs in **demo mode** against localStorage — there is
  no production data in SQLite. `prisma/dev.db` holds only seed + test rows.
- SQLite migration SQL is dialect-incompatible with Postgres; Prisma migrations cannot
  be replayed across providers anyway.

## Steps

1. **Provision Postgres.**
   - Dev/staging: Neon (recommended — free tier, branching) or Supabase.
   - Local fallback (optional): `docker run -d -p 5432:5432 -e POSTGRES_PASSWORD=dev -e POSTGRES_DB=servisaku postgres:16`.
2. **Swap the schema.** Replace `prisma/schema.prisma` with `prisma/postgres/schema.prisma`
   (keep a git commit boundary here).
3. **Point `DATABASE_URL`** in `.env` at the Postgres instance, e.g.
   `postgresql://user:pass@host/servisaku?sslmode=require`.
4. **Reset migrations.** Delete `prisma/migrations/` and `prisma/dev.db`, then:
   ```
   npx prisma migrate dev --name init_postgres
   ```
5. **Extend the seed.** `prisma/seed.js` keeps its users/coupons/bookings and gains
   catalog seeding that reads from the existing JS sources (this is the Phase 1
   catalog-unification step):
   - `src/lib/catalog/{cleaning,ac,plumbing}.js` → `ServiceCategory`/`Subcategory`/`Service`
     (+ packages as multiplier-style `ServicePackage`s)
   - `src/lib/packageData.js` `CATEGORY_PACKAGES`/`CATEGORY_ADDONS` → absolute-price
     `ServicePackage`s and `ServiceAddon`s (incl. electrical, painting, pest-control)
   - `src/lib/bookingEngine.js` `BEDROOM_OPTIONS`/`AREA_SCALED_CATEGORIES` →
     `pricingModel: 'area_based'` + `pricingConfig.tiers` on the relevant services
6. **Run the verification suite:** `node prisma/seed.js && node server/index.js` then
   `node scripts/phase0-smoke.mjs` — all 39 checks must pass unchanged (the routers are
   provider-agnostic; only `applicableServices`/`photos` parsing changes from
   string-split to `Json`, adjust `server/routes/coupons.js` + `reviews.js` mapOut).
7. **Deploy.** Host the API (Railway/Render/Fly) with `DATABASE_URL`, `JWT_SECRET`,
   `CORS_ORIGIN=https://<netlify-domain>`, `NODE_ENV=production`; add the Netlify
   `/api/*` proxy redirect so the frontend's same-origin fetch + backend detection work
   unchanged.

## Code changes required alongside (small, listed for the PR)

|------|--------|

| `server/routes/reviews.js` | `photos` becomes `Json` (drop `JSON.parse`/`stringify`) |

## Rollback

Before step 2, tag the repo (`pre-postgres`). SQLite setup is fully restorable from git
(schema + migrations + `seed.js`); `dev.db` is regenerated by `migrate dev` + seed.
